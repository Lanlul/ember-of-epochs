## 一、專案協作概覽 (Overview)

本專案的目標是打造一款 **LLM 驅動敘事、Diffusion Model 即時繪圖的文字 RPG 遊戲**。開發流程由使用者與 opencode build agent 反覆協作完成，橫跨三個主要階段：

| 階段 | 內容 | 產出 |
|------|------|------|
| **提案** | 產品定位、世界觀設計、技術選型 | `README.md` |
| **架構設計** | 系統架構圖、API 格式、任務拆解 | `ARCHITECTURE.md` |
| **實作疊代** | 7 項任務依序實作 + 無數次除錯與優化循環 | 完整 `app/` 後端 + `frontend/` 前端 + `services/` 核心邏輯 |

Agent 在過程中扮演多種角色：**產品經理**（提案）、**軟體架構師**（系統設計）、**全端工程師**（實作）、**遊戲企劃**（機制設計）、**文案指導**（品質優化）、**效能工程師**（加速），以及 **Technical Writer**（文件產生）。

## 二、關鍵 Prompt 紀錄 (Key Prompts)

### 2.1 專案啟動 Prompt

作為一切開發的起點，要求 Agent 以產品經理角色產出專案提案：

> 你現在是一位資深的 AI 產品經理。我要做一個「文字 RPG 遊戲」，遊戲能根據玩家的選擇推進劇情，並即時生成符合當下情況的情境圖片。故事背景請發揮創意自行設計。
>
> 請幫我在目前的資料夾中生成一份符合 Markdown 格式的專題提案，並自動存成 README.md 檔案。
>
> 專題需符合以下核心技術要求：
>
> - Large Language Models (LLMs)：用於生成劇情與選項，需涵蓋 Prompt Engineering。
> - Diffusion / Flow Matching Models：用於生成情境圖片，需涵蓋 Pipeline 客製化。
>
> 提案內容請包含：專題名稱與背景設定、核心解決的問題與創新點、核心功能條列、高階技術堆疊說明。

### 2.2 架構設計 Prompt

要求 Agent 轉換為軟體架構師角色，產出系統架構設計：

> 你現在是一位資深軟體架構師。請閱讀剛剛生成的 README.md，我們現在要進行「階段二：架構設計與任務拆解」。
>
> 請幫我產出一份 ARCHITECTURE.md，內容需包含：系統架構圖、API 端點設計、資料流說明、任務拆解清單（T1–T7）。

### 2.3 全面後端重構 Prompt

當遊戲遇到故事跳針、選項矛盾、圖片與故事脫節等根本性問題時，使用者下達了全面重構指令：

> 請幫我全面重構目前的後端邏輯與 LLM 串接機制。我們需要「換用更聰明的模型」加上「動態導演機制」。請依序完成以下四個實作：
>
> 1. **升級本地模型設定**：將預設模型從 3B 替換為 7B–9B 級別。
> 2. **擴充 API 參數**：新增 `plot_directive` 參數，設定 `temperature=0.65`、`repetition_penalty=1.1–1.15`、`max_tokens=1024`。
> 3. **實作 Mega-Prompt 模板**：包含角色注入（頂級小說家）、世界觀、劇情指令等結構化區塊。
> 4. **引入 Visual Prompt 機制**：模型輸出 `<visual_prompt>` 標籤，讓文字模型兼任攝影指導。

### 2.4 三部曲架構擴充 Prompt

將遊戲從單一章節擴展為三幕劇結構：

> 目前我們的 FastAPI 專案已經有了「5 個 Stage 構成一個完整故事」的進度控制邏輯。現在我需要將遊戲規模擴大為**「三部曲（3 Chapters）架構」**。
>
> 總結構為：3 個章節 (Chapter) × 5 個階段 (Stage) = 總共 15 個重大劇情節點。
>
> 核心痛點：章節切換時，絕對不能讓玩家感覺「故事斷層」或「記憶被清空」。前一章的結尾與下一章的開頭必須完美縫合。

### 2.5 隨機事件系統 Prompt

為提升可重玩性而引入隨機事件機制：

> 為了大幅提升遊戲的「可重玩性」與「故事多樣性」，我需要在現有的主線框架下，引入**「隨機突發事件系統」**。
>
> 1. 建立「章節專屬事件池」
> 2. 定義觸發機率與條件
> 3. 將隨機事件注入 User Prompt

### 2.6 效能優化 Prompt

針對生成速度過慢的問題：

> 請幫我修改後端與前端的連線機制，在「絕對不犧牲生成品質」的前提下，大幅提升推論與回應速度。
>
> 1. **串流輸出**：將 LLM API 改為 `stream=True`，使用 SSE 即時推送文字。
> 2. **視覺 Prompt 非同步生成**：故事文字完成後，背景非同步生成圖片。
> 3. **KV Cache 共享**：文字與視覺 Prompt 共享同一段推理的 KV Cache。

### 2.7 空選項除錯 Prompt

後期遇到的關鍵錯誤排除對話：

> choices = [] 代表甚麼？（使用者）
>
> 我的遊戲就是還沒到最後一回合就崩掉。（使用者）
>
> 要加，要幫我判斷是否為最後一回合。（使用者）

### 2.8 章節截斷修復 Prompt

> 剛剛玩的時候 10 回合就結局了，改成 30 回合結局。（使用者）

## 四、Agent 協助解決之技術問題 (Technical Challenges Resolved)

### 4.1 Big Pickle 模型無法載入

**問題**：啟動 Gradio 後圖片區域顯示 "bigpickle model not loaded"。

**分析**：Agent 檢查發現模型載入路徑設定錯誤，且未正確處理模型初始化失敗的狀況。

**解決方案**：
- 修正模型載入路徑與快取機制
- 加入模型載入狀態檢查與明确的錯誤訊息
- 確保 Pipeline 在應用啟動時完成初始化

### 4.2 圖片與故事內容完全脫節

**問題**：圖片雖成功生成，但內容與當前故事場景毫無關聯（例如故事在室內探索，圖片卻生成城堡外觀）。

**分析**：Agent 追蹤資料流發現，生圖 Prompt 僅包含章節層級的靜態描述，完全未參考當前回合的具體劇情內容。

**解決方案**：
- 引入 `<visual_prompt>` 標籤機制，讓文字模型在輸出故事後，根據**最後一個畫面**生成專屬繪圖提示詞
- 制定繪圖 Prompt 規則：全英文、逗號分隔、包含主體特徵、服裝、動作、背景環境與光影風格
- 將視覺 Prompt 從 User Prompt 中分離，由 LLM 動態生成而非固定模板

### 4.3 故事跳針與內容重複

**問題**：故事持續在同一場景迴圈（如「走來走去看到符文、書櫃」），選擇不影響後續發展，句子結構機械化。

**分析**：Agent 比對多回合輸出發現：
- 3B 模型容量不足，無法維持連貫敘事
- 缺乏 `plot_directive` 機制，LLM 不知道每回合應推進到哪個劇情點
- 缺乏 `repetition_penalty` 參數，模型傾向重複已生成的詞彙

**解決方案**：
- **模型升級**：從 3B 替換為 7B–9B 級別模型
- **動態導演機制**：新增 `plot_directive` 參數，每回合告訴 LLM「這段必須達成的劇情點」
- **參數調校**：設定 `temperature=0.65`、`repetition_penalty=1.15`、`max_tokens=1024`
- **結構化 Mega-Prompt**：設計包含角色注入、世界觀、歷史摘要、劇情指令的標準模板

### 4.4 三部曲架構實作與章節銜接

**問題**：從 5 個 Stage 擴展為 3 Chapters × 5 Stages = 15 節點時，章節切換處出現故事斷層。

**分析**：Agent 檢視發現 Stage 配置僅為線性陣列，缺乏章節歸屬資訊，且章節結尾的 `plot_directive` 未特別設計。

**解決方案**：
- 建立巢狀的 Chapters → Stages 配置結構
- 每章節包含獨立標題、世界觀（continents）、目標說明
- Chapter 結尾的 Stage 5 使用「懸念式」plot_directive（禁止總結，引導至下一章）
- 僅 Chapter 3 Stage 5 使用「最終結局」directive

### 4.5 隨機事件與場景衝突

**問題**：隨機抽取的具體事件（如沙塵暴）與前一回合場景（如室內）產生邏輯矛盾。

**分析**：Agent 發現事件池使用具體描述（"沙塵暴阻擋去路"），而非抽象母題，LLM 被迫在矛盾中生成故事。

**解決方案**：
- 將事件池從「具體事件」重構為「抽象戲劇母題」：環境危機、不速之客、資源匱乏、身心狀況、隱藏資訊
- 將抽象母題注入 Prompt，讓 LLM 自行決定具體表現形式，確保與當前場景一致

### 4.6 Generation 參數錯誤導致崩潰

**問題**：`chronicle.py` 呼叫 `generate()` 時出現 `takes 2 positional arguments but 3 were given` 錯誤。

**分析**：Agent 定位發現 `generate()` 方法簽章為 `generate(self, messages)`，但呼叫端傳入了 `generate(system_prompt, user_prompt)` 兩個位置引數。

**解決方案**：將兩個 Prompt 包裝為標準的 `[{role: "system", content: ...}, {role: "user", content: ...}]` 格式後傳入。

### 4.7 空 Choices 導致遊戲崩潰

**問題**：LLM 輸出的 `<options>` 內 JSON `choices` 陣列為空，前端出現「請先選擇一個選項」錯誤且無法繼續。

**分析**：Agent 分析發現 `_finalize_turn` 在解析 LLM 輸出時，若 JSON 的 `choices` 為空陣列但 `<options>` 標籤內有文字選項，未建立後備的 Choice 物件。

**解決方案**：
- 在 `_finalize_turn` 中加入 `else` 分支：當 JSON choices 為空時，從 `<options>` 文字的選項列點手動建立 Choice 物件
- 在 `process_action` 中加入重試迴圈：若 choices 為空且非最終階段，最多重試 2 次 LLM 呼叫
- 重構 `_finalize_turn` 使其可接受預解析的 `parsed` 參數，避免重複解析

### 4.8 CHAPTERS 設定檔被截斷

**問題**：遊戲在第 10 回合就觸發結局，而非預期的 30 回合。

**分析**：Agent 檢查 `script_config.py` 發現 `CHAPTERS` 串列因不明原因被截斷，僅剩第一章（5 個 Stages），`total_turns()` 回傳 `1 × 10 = 10`，第 10 回合即滿足 `is_final_stage` 條件（`global_turn >= total_turns()`）。

**解決方案**：
- 補回第二章（翡翠密境／永凍星環）與第三章（虛空迴廊）的完整 Stage 配置
- 驗證 `total_turns()` 正確回傳 30，`is_final_stage(10)` 回傳 False

### 4.9 開場選擇不影響第一回合

**問題**：遊戲狀態顯示「回合 1 | Stage 1」出現兩次，且開場的選擇似乎未影響後續故事。

**分析**：Agent 檢查 `_prepare_turn` 發現當 `history` 為空時（開場選擇後第一回合），無法從歷史記錄中找到玩家選擇的文字，導致劇情上下文遺失。

**解決方案**：
- 新增 `_find_opening_choice_text()` 方法，從開場流程中提取玩家選擇
- 將狀態標籤從「回合 1」調整為「序幕」，明確區分開場與正式回合
